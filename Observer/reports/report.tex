\section{Real-world Problem}
Consider a modern banking application where users perform transaction operations such as deposits on their bank accounts. Whenever a transaction occurs (e.g., a deposit is made), the system needs to notify various downstream services to process this event:
\begin{itemize}
    \item Update the user interface (UI) to reflect the new balance.
    \item Send an email notification/receipt to the user.
    \item Log the transaction details for auditing and record-keeping.
    \item Run real-time fraud detection analytics to ensure transaction safety.
    \item Send a mobile push notification to the user's phone.
    \item Forward data to analytics systems for user behavior reporting.
\end{itemize}

The core challenge is establishing a system where a bank account can broadcast transaction updates to a dynamic set of interested services/observers without tightly coupling the core banking logic to concrete service classes or notification mechanisms.

\section{Naive Solution (Without Design Pattern)}
In a naive implementation, the \texttt{NaiveBankAccount} class directly references and invokes methods on concrete service classes whenever a deposit is made. Below is the C++ representation of this approach:

\begin{lstlisting}[caption=Naive Implementation of Bank Account]
// Naive implementation demonstrating tight coupling
class NaiveBankAccount {
private:
    double balance;
    std::string accountNumber;
    
    // Direct dependencies on concrete notification/processing services
    UIUpdateService* uiService;
    EmailService* emailService;
    LoggerService* loggerService;
    FraudDetectionService* fraudService;
    MobileAppService* mobileService;
    AnalyticsService* analyticsService;

public:
    NaiveBankAccount(std::string accNum) 
        : balance(0.0), accountNumber(accNum),
          uiService(new UIUpdateService()),
          emailService(new EmailService()),
          loggerService(new LoggerService()),
          fraudService(new FraudDetectionService()),
          mobileService(new MobileAppService()),
          analyticsService(new AnalyticsService()) {}

    ~NaiveBankAccount() {
        delete uiService; delete emailService; delete loggerService;
        delete fraudService; delete mobileService; delete analyticsService;
    }

    void deposit(double amount) {
        balance += amount;
        
        // Direct method invocation on concrete objects
        uiService->updateUI(amount, balance);
        emailService->sendEmail(accountNumber, amount);
        loggerService->logTransaction(accountNumber, amount);
        fraudService->detectFraud(accountNumber, amount);
        mobileService->pushNotification(amount);
        analyticsService->trackEvent("deposit", amount);
    }
};
\end{lstlisting}

\section{Problems with the Naive Solution}
The naive approach exhibits several major architectural flaws:
\begin{enumerate}
    \item \textbf{Tight Coupling:} The \texttt{NaiveBankAccount} is tightly coupled to concrete classes like \texttt{UIUpdateService}, \texttt{EmailService}, etc. It must hold references to each concrete service and know exactly which methods to invoke on them.
    \item \textbf{Violation of the Open-Closed Principle (OCP):} If we want to introduce a new downstream service (such as a regulatory auditing service or a tax calculator), we must modify the \texttt{NaiveBankAccount} class. This requires adding a new pointer, allocating it in the constructor, deleting it in the destructor, and invoking it inside \texttt{deposit}.
    \item \textbf{Violation of the Single Responsibility Principle (SRP):} The bank account class should focus solely on managing balance and basic transaction logic. Instead, it is tasked with orchestrating notification emails, logging, UI rendering, and fraud checks.
    \item \textbf{Rigidity:} Since the dependencies are hardcoded, they cannot be configured or swapped at runtime, making unit testing difficult (cannot mock dependencies).
\end{enumerate}

\section{Introduction to the Observer Pattern}
The \textbf{Observer Pattern} is a behavioral design pattern that defines a one-to-many dependency between objects. When one object (the \textbf{Subject} or \textbf{Publisher}) changes state, all its dependents (the \textbf{Observers} or \textbf{Subscribers}) are notified and updated automatically.

Key concepts:
\begin{itemize}
    \item \textbf{Subject:} Knows its observers and provides an interface to attach or detach them.
    \item \textbf{Observer:} Defines an updating interface for objects that should be notified.
    \item \textbf{ConcreteSubject:} Stores state of interest to ConcreteObservers and sends notifications.
    \item \textbf{ConcreteObserver:} Implements the Observer updating interface to keep its state consistent with the subject.
\end{itemize}

\section{Implementation of the Observer Pattern}
By decoupling the classes through abstract interfaces, our refactored implementation allows adding new service observers at runtime without modifying the subject class. We also utilize modern C++ smart pointers (\texttt{std::shared\_ptr}) to safely manage object lifetimes.

\begin{lstlisting}[caption=Refactored Observer Implementation]
// Observer interface
class TransactionObserver {
public:
    virtual ~TransactionObserver() = default;
    virtual void update(double amount, double balance) = 0;
};

// Concrete Observers
class UIObserver : public TransactionObserver {
public:
    void update(double amount, double balance) override {
        std::cout << "[UI] Updated. Transaction: " << amount 
                  << ", New Balance: " << balance << "\n";
    }
};

class EmailObserver : public TransactionObserver {
public:
    void update(double amount, double balance) override {
        std::cout << "[Email] Sent receipt. Amount: " << amount 
                  << ", Current Balance: " << balance << "\n";
    }
};

// Subject / Publisher
class BankAccount {
private:
    double balance = 0.0;
    std::string accountNumber;
    std::vector<std::shared_ptr<TransactionObserver>> observers;

public:
    BankAccount(std::string accNum) : accountNumber(accNum) {}

    void attach(std::shared_ptr<TransactionObserver> observer) {
        observers.push_back(observer);
    }

    void detach(std::shared_ptr<TransactionObserver> observer) {
        observers.erase(
            std::remove(observers.begin(), observers.end(), observer),
            observers.end()
        );
    }

    void deposit(double amount) {
        balance += amount;
        notify(amount);
    }

    void notify(double amount) {
        for (const auto& observer : observers) {
            observer->update(amount, balance);
        }
    }
};
\end{lstlisting}

\section{Pros and Cons}
\subsection{Pros}
\begin{itemize}
    \item \textbf{Loose Coupling:} The Subject and Observers are loosely coupled. The Subject only knows that the observer implements a specific interface.
    \item \textbf{Support for Broadcast Communication:} Unlike direct calls, notifications are broadcast automatically to all registered observers.
    \item \textbf{Open-Closed Principle:} We can introduce new concrete observer classes without changing the subject's code.
\end{itemize}

\subsection{Cons}
\begin{itemize}
    \item \textbf{Lapsed Listener Problem:} Observers must be carefully deregistered when destroyed, or they remain in memory and get called, causing undefined behavior or leaks.
    \item \textbf{Unintended Update Chains:} If observers trigger updates on subjects, it can lead to complex and hard-to-debug cascading updates or infinite loops.
    \item \textbf{Ordering Concerns:} There is no guarantee in which order the observers will be notified.
\end{itemize}

\section{Other Real-world Applications}
\subsection{Web Development}
\begin{itemize}
    \item \textbf{Event Handling in JavaScript:} The DOM's \texttt{addEventListener} uses the Observer pattern to attach event handlers to HTML elements.
    \item \textbf{State Management (Redux/Vuex):} Views subscribe to changes in a global store and re-render automatically when state mutations occur.
\end{itemize}
\subsection{Mobile Development}
\begin{itemize}
    \item \textbf{RxSwift / RxJava / RxKotlin:} Reactive streams rely heavily on observing data sequences and binding them directly to UI components.
    \item \textbf{SwiftUI/Android Jetpack Compose:} Declarative UI frameworks use property wrappers like \texttt{@Published} and state flows to push model updates directly to the view.
\end{itemize}
